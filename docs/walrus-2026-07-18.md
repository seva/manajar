# Session 2 — Phase 2 Vertical Slice

## Wins
- Bootstrap scripts (server + client) created, 7 modules load in edit mode
- `default.project.json` fixed: `$className` added to `StarterPlayer` and `StarterPlayerScripts`
- `$className "StarterPlayerScripts"` confirmed in built `.rbxlx` XML
- PyAutoGUI (0.9.54) works for programmatic F5: `win.activate()` + `pyautogui.press('f5')`
- Play mode starts: `PlaySoloSuccess` confirmed in Studio log

## Learnings
- Roblox Studio `warn()` output goes to Output window, NOT to `%LOCALAPPDATA%\Roblox\logs\*Studio*` log files
- `pyautogui` (via `pygetwindow.win.activate()`) worked for F5 where PowerShell SendKeys/PostMessage/SendInput/AppActivate all failed
- `.NET UIAutomation` can't find Ribbon Play button — Studio uses custom Ribbon framework
- After clicking to focus window, keyboard shortcuts must be sent immediately — window loses focus if you `time.sleep()` between clicks and keys
- `pyperclip.paste()` returned empty after Ctrl+C in Studio — clipboard may need explicit `time.sleep()` or Studio's Output window may not have been focused

## Risks
- Can't capture Output window content programmatically — `pyperclip` returns empty
- Studio crashes/kills when F5 is sent to the wrong window
- Multiple Studio instances accumulate and fight for port/file locks

## Strategy
- Use pygetwindow to find Studio window by title, activate it, then pyautogui for all input
- For capturing Output: click Output tab (bottom panel), Ctrl+A, Ctrl+C in rapid succession without delays
- After each `rojo build`, kill ALL `RobloxStudio*` processes before starting fresh
- Or use `rojo serve` + Studio's live sync to avoid rebuild/open cycle

## Decisions
- `$className` is required for `StarterPlayer` and `StarterPlayerScripts` in Rojo 7 `default.project.json` — without it, Rojo creates `Folder` instances and LocalScripts never replicate to players
- `rojo build` is preferred over `rodeo run --place` for generating place files (rodeo npm package errors with "No message was specified")

## Tradeoffs
- `rojo build` + `Start-Process` gives a clean session but requires rebuild on every code change
- `rojo serve` gives live sync but Studio auto-reload may miss updates
- `win.activate()` + `pyautogui.press('f5')` is reliable but pyautogui automation is fragile if the window position changes

## Alternatives
- Could use `rojo serve` + Studio plugin for auto-deploy on save
- Could investigate Studio's `-play` command-line switch for automatic play mode startup
- Could write a Studio plugin that logs to ReplicatedStorage for programmatic reading

# Session 4 — Gameplay Verification, Headless Test Runner, Constitutional Compliance

## Wins
- Full gameplay loop verified programmatically: fill jar → select target → cast spell → server processes → client receives UpdateJarDisplay
- Rodeo CLI (v1.1.0) at `C:\Users\seval\AppData\Local\Temp\rodeo\rodeo.exe` — headless test runner working
- 9/9 runnable test files pass, 0 failures (was 3 failures at start of session)
- Lighting migration warning eliminated (`Technology: Voxel` in project.json)
- ClientHandler type syntax error fixed (bracket notation → named type alias)
- Replication timing fixed (`WaitForChild` for ClientHandler module)
- Commit history cleaned (bad message format fixed via detached HEAD + cherry-pick)
- Constitutional audit: all critical violations resolved

## Learnings
- **Colon vs dot notation is the #1 Roblox Luau footgun.** `handler.remoteEvents.FillJar:FireServer(amount)` passes `self` implicitly. Wrapper functions must declare `function(self, amount)` to absorb it, not `function(amount)`. Same applies to test mocks: if the module calls `jarSystem.FillJar(jar, amount)` (dot), the mock must be `FillJar = function(jar, amount)`, NOT `function(_, jar, amount)`.
- **`ClientHandler["remoteEvents"]` bracket notation is invalid Luau syntax.** Extract inline table types to named `export type` aliases.
- **ModuleScript replication is not instant.** `script.Parent.ClientHandler` can fail if the module hasn't replicated yet. Always use `script.Parent:WaitForChild("ClientHandler")` in client scripts.
- **Rodeo CLI** runs Luau scripts against a place file in Studio with full Roblox API. Modes: `edit|run|test|play`. Contexts: `plugin|server|client|elevated`. Plugin context in edit mode is simplest for test runners.
- **TestEZ-style `describe`/`it`/`expect` are NOT globals in Studio.** They must be provided by a framework module or passed as parameters to test functions.
- **`expect(table).toEqual({})` fails** because `toEqual` compares table references, not structure. Use `expect(next(table)).toEqual(nil)` for emptiness checks.
- **`git rebase -i` opens an editor and times out in pwsh.** Workaround: `git checkout <bad-commit>`, `git commit --amend`, then `git cherry-pick` subsequent commits, then `git branch -f main` + force push.
- **pyautogui inputs don't reach Studio viewport without a click.** After F5, must `pyautogui.click(center_x, center_y)` before sending game inputs.
- **`default.project.json` needs `"Technology": "Voxel"`** under Lighting service to prevent Studio's auto-migration warning from Compatibility Lighting.

## Risks
- 2 test files skipped (SetupRemotes_spec, GameManager_spec) — they monkey-patch `Instance.new` and `_G.game`, which are read-only in Studio. Need rewrite to use real Roblox instances or a different mocking strategy.
- Test runner is a custom implementation, not TestEZ. Missing features: `beforeEach`/`afterEach`, nested describe scoping, `toThrow` with message matching.
- Headed test Output window capture via clipboard remains unreliable.

## Strategy
- Use Rodeo CLI for all headless testing: `rojo build` → `rodeo run --place --mode edit --context plugin --show-return test-runner.luau`
- Keep test-framework.luau minimal — add features only as tests need them
- Rewrite skipped tests to work in Studio (use real RemoteEvents, real Instance hierarchy)

## Decisions
- Test runner lives at project root (`test-runner.luau` + `test-framework.luau`), not in `src/`
- Tests are included in Rojo build under `ReplicatedStorage.tests` (not a separate service)
- `describe`/`it`/`expect` passed as parameters to test functions, not set as globals (avoids pollution)
- Lighting Technology set to Voxel (not ShadowMap or Future) — simplest, kid-friendly, no migration warning

## Tradeoffs
- Custom test framework vs TestEZ: simpler, no dependency, but missing features. Acceptable for current scope.
- Tests in ReplicatedStorage vs separate TestService: simpler Rojo config, but tests ship with the game. Acceptable for now — can exclude later.
- Rodeo plugin context vs server/client context: plugin context has full access but doesn't test replication. Good for unit tests, not integration.

## Alternatives
- Could use `jest_rodeo` (pesde package) for Jest-style testing via Rodeo
- Could use TestEZ via wally package manager (standard Roblox approach)
- Could use `rodeo run --mode test --context server` for server-side integration tests
- Could write a Studio plugin that auto-runs tests on save
