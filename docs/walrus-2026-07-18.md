# Session 2 — Phase 2 Vertical Slice

## Wins
- Bootstrap scripts (server + client) created, 7 modules load in edit mode
- `default.project.json` fixed: `$className` added to `StarterPlayer` and `StarterPlayerScripts`
- `$className "StarterPlayerScripts"` confirmed in built `.rbxlx` XML
- PyAutoGUI (0.9.54) works for programmatic F5: `win.activate()` + `pyautogui.press('f5')`
- Play mode starts: `PlaySoloSuccess` confirmed in Studio log

## Learnings
- Roblox Studio `warn()` output goes to Output window, NOT to `%LOCALAPPDATA%\Roblox\logs\*Studio*` log files
- `pyautogui` (via `pygetwindow.win.activate()`) worked for F5 where PowerShell SendKeys/PostMessage/SendInput/AppActivate all failed
- `.NET UIAutomation` can't find Ribbon Play button — Studio uses custom Ribbon framework
- After clicking to focus window, keyboard shortcuts must be sent immediately — window loses focus if you `time.sleep()` between clicks and keys
- `pyperclip.paste()` returned empty after Ctrl+C in Studio — clipboard may need explicit `time.sleep()` or Studio's Output window may not have been focused

## Risks
- Can't capture Output window content programmatically — `pyperclip` returns empty
- Studio crashes/kills when F5 is sent to the wrong window
- Multiple Studio instances accumulate and fight for port/file locks

## Strategy
- Use pygetwindow to find Studio window by title, activate it, then pyautogui for all input
- For capturing Output: click Output tab (bottom panel), Ctrl+A, Ctrl+C in rapid succession without delays
- After each `rojo build`, kill ALL `RobloxStudio*` processes before starting fresh
- Or use `rojo serve` + Studio's live sync to avoid rebuild/open cycle

## Decisions
- `$className` is required for `StarterPlayer` and `StarterPlayerScripts` in Rojo 7 `default.project.json` — without it, Rojo creates `Folder` instances and LocalScripts never replicate to players
- `rojo build` is preferred over `rodeo run --place` for generating place files (rodeo npm package errors with "No message was specified")

## Tradeoffs
- `rojo build` + `Start-Process` gives a clean session but requires rebuild on every code change
- `rojo serve` gives live sync but Studio auto-reload may miss updates
- `win.activate()` + `pyautogui.press('f5')` is reliable but pyautogui automation is fragile if the window position changes

## Alternatives
- Could use `rojo serve` + Studio plugin for auto-deploy on save
- Could investigate Studio's `-play` command-line switch for automatic play mode startup
- Could write a Studio plugin that logs to ReplicatedStorage for programmatic reading
